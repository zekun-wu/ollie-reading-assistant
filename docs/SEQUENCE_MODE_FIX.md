# 🔧 Sequence Mode Cache Fix - Eye-Tracking Issue Resolved

## Problem Identified

**Symptom**: 
- ✅ `time.json` saved correctly to `backend/mixed/1/time.json`
- ❌ Eye-tracking files saved with wrong names:
  - `2_eye_main_aoi_4.wav` instead of `1_que_2_4_main.wav`
  - `2_eye_asst_aoi_4.json` instead of `1_que_2_4_eye_asst.json`

**Root Cause**: 
Eye-tracking guidance is generated by `state_manager.py`, which was NOT receiving or passing `sequence_step` to TTS and cache services.

---

## The Missing Chain

### Flow Analysis

**Time Tracking** (Working ✅):
```
Frontend → API → TimeTrackingService (with sequence_step) → SequenceCacheService ✅
```

**Eye-Tracking** (Was Broken ❌):
```
Frontend → WebSocket → StateManager (NO sequence_step) → TTS/Cache Services ❌
```

### Why It Was Broken

The eye-tracking flow is different from manual assistance:
1. **Manual Assistance**: Direct HTTP API calls from hooks
2. **Eye-Tracking**: WebSocket messages + State Manager orchestration

The State Manager was missing:
1. `sequence_step` field in `SessionState` dataclass
2. `sequence_step` parameter in `start_tracking()` method
3. Extraction of `sequence_step` from WebSocket messages
4. Passing `sequence_step` to TTS and cache service calls

## Verification of Fix

### Complete Chain (Now Working ✅)

**Sequence Mode - Eye-Tracking Assistance**:

1. `SequenceReader` extracts `sequenceStep = currentStepIndex + 1` (e.g., 1)
2. `useGazeState` hook receives `sequenceStep = 1`
3. When `startTracking()` is called, adds `sequence_step: 1` to WebSocket message
4. Backend WebSocket handler receives `{"type": "start_tracking", "sequence_step": 1, ...}`
5. State Manager extracts `sequence_step = 1` from message
6. Creates `SessionState` with `sequence_step = 1`
7. When AOI triggers guidance, `_generate_llm_curiosity_guidance()` is called
8. Passes `session.sequence_step = 1` to TTS service
9. TTS service lazy-loads `SequenceCacheService`
10. Generates path: `backend/mixed/1/1_que_2_4_main.wav`
11. Saves file to correct location ✅
